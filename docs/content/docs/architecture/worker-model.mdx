# Worker Model

Strategies do not run inside the renderer or directly inside the Electron main thread.

They run in Node worker threads managed by the host.

## Worker Ownership

In the current implementation, workers are keyed by `conversationId`.

That means Looma maintains a worker runtime per active conversation, not one global worker per enabled strategy.

This matters for state and lifecycle expectations.

## What Runs in the Worker

The worker is responsible for:

- loading the strategy module
- constructing the `ctx` object
- running strategy hooks
- sending host requests for history, memory, state, LLM calls, and tool execution
- sending dev-inspector events back to the host

The worker does not talk to SQLite or provider SDKs directly.

## Host-Worker Communication

The host and worker communicate through structured message passing.

The host can request:

- `contextBuild`
- `turnEnd`
- `toolCall`
- `replayTurn`
- `dispose`

The worker can request host services such as:

- history lookup
- current turn input lookup
- memory queries and asset reads
- state get/set/delete
- host-side tool execution
- host-side LLM calls and loop runs

## Timeouts

The worker manager currently applies per-request timeouts.

Notable timeout classes include:

- context build
- turn end
- replay turn
- tool calls, with tool-specific overrides for web search and web fetch

Timeout handling is host-owned.

## Cleanup and Restart

Workers are disposed when the host explicitly tears them down for a conversation.

If a worker becomes unhealthy or times out on a request that should terminate it, the host can terminate that worker and recreate it later.

Because of that, strategy authors should treat in-memory worker state as ephemeral.
