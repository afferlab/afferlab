# Strategy Engine

The strategy engine is the programmable layer that decides how a conversation prompt is built before the model call runs.

In the current implementation, a strategy is a TypeScript module loaded by the strategy worker runtime.

## What a Strategy Exports

Looma currently supports these exports:

- `meta`
- `configSchema`
- `onContextBuild`
- `onInit`
- `onTurnEnd`
- `onReplayTurn`
- `onCleanup`
- `onError`
- `onToolCall`

Only `onContextBuild` is required.

## What `onContextBuild` Does

`onContextBuild(ctx)` is the main hot path.

It receives the current strategy context and must return:

```ts
{
  prompt: { messages: Message[] },
  tools?: ToolDefinition[],
  meta?: { ... }
}
```

In practice, built-in strategies build prompts through `ctx.slots.add(...)` and then return `ctx.slots.render()`.

## Host Responsibilities

The Electron host still owns critical runtime behavior:

- loading the active strategy for a conversation
- building the worker request payload
- fetching history and current input
- running the actual LLM stream
- executing host-owned tools
- persisting messages and turn state

So the strategy engine is programmable, but it is not a fully hostless agent runtime.

## Built-In Strategies Today

The current built-in strategies seeded by the host are:

- `Base`
- `Cloud`

`Base` is the minimal chat strategy.

`Cloud` enables the Memory Cloud flow and reads pinned assets into the prompt.

## Turn Lifecycle

At a high level, a normal turn works like this:

1. The host creates the user turn and assistant placeholder.
2. The host asks the worker to run `onContextBuild`.
3. The worker builds prompt messages and optional tool definitions.
4. The host runs the LLM stream.
5. After the final assistant message is written, the host may call `onTurnEnd`.

If strategy context building fails, the host currently falls back to raw host history instead of failing the entire send flow.
