# System Overview

Looma currently runs as a desktop application with a React renderer and an Electron backend.

The backend is not a single process blob. It is split into a host runtime, worker runtime, IPC boundaries, and local infrastructure services.

## Runtime Layers

At runtime, Looma is organized into four practical layers:

- `src/`: the renderer UI
- `electron/ipc`: the renderer-to-host boundary
- `electron/engine`: host-side application logic such as chat, strategy, settings, and streaming
- `electron/workers/strategy`: isolated strategy execution

Supporting infrastructure lives under:

- `electron/infra/db`: SQLite connection, schema, and vector helpers
- `electron/llm/providers`: provider adapters
- `electron/preload`: the safe bridge exposed to the renderer

## High-Level Flow

The main chat path is:

1. The renderer sends a request through preload.
2. IPC handlers validate and route the request.
3. `engine/chat/application` creates or updates turn state.
4. `engine/strategy/host` asks a strategy worker to build prompt context.
5. `engine/chat/streaming/StreamManager` calls `engine/llm/llmRunner`.
6. Stream chunks are sent back to the renderer.

This is the live runtime path in the current codebase.

## Local-First Boundaries

Looma is local-first in the sense that conversation state, strategy state, settings, and memory metadata are stored locally.

Today that storage is split between:

- SQLite tables for conversations, turns, messages, settings, strategies, and memory metadata
- local file-backed assets for uploaded documents and other attachments

So the current implementation is local-first, but it is not a single-file storage model.
